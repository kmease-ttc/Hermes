Here’s a “developer playbook” so every current and future page stays visually consistent with the new home page.

I’ll organize it into:

1. global layout & theme,
2. reusable components,
3. section patterns,
4. page templates,
5. checklist for any new page.

You can paste this straight into your dev doc / README.

---

## 1. Global Layout & Theme

### 1.1 App shell

* Use a single layout component for all marketing pages, e.g. `AppLayout`:

  * Includes:

    * Top tagline bar: “BUILT FOR PEOPLE WHO DON’T WANT TO THINK ABOUT SEO”
    * Glass nav pill
    * Main content container
    * Footer

  ```tsx
  export function AppLayout({ children }: { children: React.ReactNode }) {
    return (
      <div className="min-h-screen bg-gradient-to-b from-white via-slate-50 to-slate-100 text-slate-900">
        <TopBanner />
        <Header />
        <main className="pb-24">{children}</main>
        <Footer />
      </div>
    );
  }
  ```

* All pages should render inside `<AppLayout>`.

### 1.2 Content width & spacing

* Max width for all content sections:

  ```tsx
  const container = "max-w-6xl mx-auto px-4 md:px-8";
  ```
* Standard vertical spacing for every section:

  ```tsx
  py-24 md:py-28
  ```

  (Use this on *every* `<section>` by default.)

### 1.3 Tailwind theme

* Always use the **named tokens** from `tailwind.config` instead of raw hex values:

  * `bg-primary`, `bg-surface`, `bg-surface-dark`, etc.
  * `text-title`, `text-body`, `text-muted`.
  * Shadows: `shadow-soft`, `shadow-hero`, `shadow-glass`.

* If you add new colors/shadows, add them to the theme and **never inline hex** in components.

## 2. Reusable Components (use these everywhere)

### 2.1 Section shell

Create a generic `Section` component so the spacing/width is never hand-rolled:

```tsx
type SectionProps = {
  children: React.ReactNode;
  className?: string;
  id?: string;
  variant?: "light" | "dark" | "gradient";
};

export function Section({
  children,
  className = "",
  id,
  variant = "light",
}: SectionProps) {
  const base = "py-24 md:py-28";
  const bg =
    variant === "dark"
      ? "bg-surface-dark text-white"
      : variant === "gradient"
      ? "bg-gradient-section"
      : "";

  return (
    <section id={id} className={`${base} ${bg} ${className}`}>
      <div className="max-w-6xl mx-auto px-4 md:px-8">{children}</div>
    </section>
  );
}
```

Use `Section` for every block (features, how it works, pricing, examples, roadmap, etc).

### 2.2 Typography components

Create small primitives so headings are consistent:

```tsx
export const Overline = ({ children }: { children: React.ReactNode }) => (
  <p className="text-xs font-semibold tracking-[0.2em] uppercase text-primary-600">
    {children}
  </p>
);

export const SectionTitle = ({ children }: { children: React.ReactNode }) => (
  <h2 className="mt-3 text-3xl md:text-4xl font-extrabold tracking-tight text-slate-900">
    {children}
  </h2>
);

export const SectionSubtitle = ({
  children,
}: {
  children: React.ReactNode;
}) => (
  <p className="mt-3 text-base md:text-lg text-slate-600 max-w-2xl">
    {children}
  </p>
);
```

Use these for every page heading instead of custom text classes.

### 2.3 Buttons

You already have a shared `Button` component – treat it as the **only** way to render buttons.

* Variants to use:

  * `variant="primary"` → gradient pill (hero, primary CTAs).
  * `variant="secondary"` → white pill with border.
  * `variant="ghost"` → text-style link buttons.

* Ensure Button always has:

  * `focus-visible:outline-none focus-visible:ring-2 focus-visible:ring-primary-500 focus-visible:ring-offset-2`.

### 2.4 Card primitives

Create a base card and extend it:

```tsx
export function Card({
  children,
  className = "",
}: {
  children: React.ReactNode;
  className?: string;
}) {
  return (
    <div
      className={`
        rounded-3xl bg-white/90 backdrop-blur
        border border-slate-200/70
        shadow-soft
        ${className}
      `}
    >
      {children}
    </div>
  );
}
```

Then build specific variants:

* `FeatureCard` (icon + title + body)
* `StatCard` (big number + label)
* `PricingCard`
* `ExampleCard` (screenshot + overlay)
* `StepCard` (How It Works)

All of them should compose `Card` instead of redefining radius, border, shadow every time.

### 2.5 Floating Tag / Pill

Use one pill style everywhere (hero tags, roadmap chips, category labels):

```tsx
export const Pill = ({
  children,
  className = "",
}: {
  children: React.ReactNode;
  className?: string;
}) => (
  <span
    className={`
      inline-flex items-center gap-2
      rounded-full bg-white/95 backdrop-blur
      px-4 py-2 text-xs font-medium text-slate-700
      border border-slate-200/80 shadow-pill
      ${className}
    `}
  >
    {children}
  </span>
);
```

## 3. Section Patterns to Reuse on Future Pages

### 3.1 Light “marketing” section

* Use `Section variant="light"`.
* Overline + SectionTitle + SectionSubtitle.
* Content in 2–3 columns of `Card` components.

### 3.2 Dark “proof / data” section

* `Section variant="dark"` with the existing gradient/background utility (`bg-surface-dark` or `bg-gradient-dark`).
* Use `GlassStatCard` variant: `bg-white/5 backdrop-blur-xl border-white/10 shadow-hero`.
* Always include:

  * Overline → “Proof it works / Results”.
  * 3–4 stat cards in `grid md:grid-cols-3`.

### 3.3 Hero pattern

If you add heroes to other pages (`/pricing`, `/examples`, `/about`):

* Always follow:

  * Left: Overline + H1 + paragraph + primary/secondary CTAs.
  * Right: Card with screenshot/mockup + floating `Pill`s.

* Use the same hero wrapper as current home hero:

  * `Card` with larger radius, `shadow-hero`, dark mockup container.

### 3.4 “Steps / Timeline” pattern

For any process explanation:

* Use the upgraded How It Works timeline pattern:

  * Vertical line on desktop, stacked cards on mobile.
  * Each step is a `StepCard` w/ `Pill` for “Step 1”, mini image, title, short description.

### 3.5 “Gallery / Examples” pattern

For case studies, integrations, templates, etc.:

* Use `ExampleCard` structure:

  * Screenshot with `hover:scale-[1.03]`.
  * Dark gradient overlay + “View…” pill on hover.
  * Category `Pill` pinned top-left of image.
  * Title + body text + optional meta row under image.

## 4. Page Templates

Have the dev create a couple of **page skeletons** and reuse them.

### 4.1 Generic marketing page template

File: `PageMarketing.tsx`

```tsx
export function PageMarketing({
  hero,
  sections,
}: {
  hero: React.ReactNode;
  sections: React.ReactNode[];
}) {
  return (
    <AppLayout>
      {hero}
      {sections.map((section, i) => (
        <Fragment key={i}>{section}</Fragment>
      ))}
    </AppLayout>
  );
}
```

Use it like:

```tsx
export default function PricingPage() {
  return (
    <PageMarketing
      hero={<PricingHero />}
      sections={[
        <PricingTableSection />,
        <FAQSection />,
        <RoadmapTeaserSection />,
      ]}
    />
  );
}
```

### 4.2 About / Story template

* Same idea: hero + alternating light sections (story, experience, vision) + dark proof section reused from home.

## 5. New Page Checklist (what dev should run through every time)

For any new page or major section, dev should confirm:

1. **Layout**

   * Page wrapped in `<AppLayout>`.
   * All content sections use `<Section>` with standard padding and container width.

2. **Typography**

   * Page titles use `SectionTitle` or `H1` from typography components.
   * Overlines use `Overline`.
   * Body text uses `text-base md:text-lg text-slate-600` via shared component/utility.

3. **Color & Shadows**

   * All colors reference Tailwind theme tokens – no stray hex codes.
   * Cards use `Card` or a variant; shadows only from `shadow-soft`, `shadow-hero`, etc.

4. **Buttons & Links**

   * All buttons use the shared `Button` component.
   * Focus states visible (keyboard-accessible).
   * CTAs follow primary/secondary pattern from hero.

5. **Spacing**

   * No custom `py-12`/`py-16` unless there is a strong reason.
   * Gap between cards and elements uses a small set of values (`gap-6`, `gap-8`, `space-y-6/8/10`).

6. **Hover / Motion**

   * Cards: subtle lift + shadow change, nothing extreme.
   * Images: max `scale-[1.03]` on hover.
   * No new animation durations/timing functions—reuse existing ones.

7. **Responsive**

   * All grids collapse to 1-column on mobile.
   * Floating pills / tags remain inside the viewport on small screens.
   * Hero mockup doesn’t overflow on narrow widths.

If your dev follows this playbook and reuses the shared building blocks (Layout, Section, Card, Button, Pill, typography components), every current and future page will automatically feel like part of the same system as the home page.
